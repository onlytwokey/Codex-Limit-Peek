# Local-Only Documentation Rendering Design

**Status:** Approved

**Date:** 2026-07-18

## Context

Codex Limit Peek generates four README PNGs from production SwiftUI and
AppKit components with fixed demonstration data. Local macOS 26 / Swift 6.3
and GitHub's macOS 15 / Swift 6.1 runner both produce structurally valid
images, but their rendered bytes differ. Exact committed-image comparison is
therefore not a portable GitHub CI contract.

Removing only the dedicated
`scripts/render-doc-previews.sh --check` workflow step would be incomplete.
The ordinary Swift test command currently runs three tests that perform
offscreen documentation rendering, including the test that writes the four
PNG assets to a temporary directory.

## Goals

- Keep source-backed documentation rendering available to contributors.
- Make local `scripts/render-doc-previews.sh` the only workflow that generates
  the four README PNGs.
- Ensure GitHub CI does not execute documentation preview rendering or compare
  locally approved assets with runner-generated bytes.
- Keep non-rendering documentation fixture, state, isolation, and seam tests
  in GitHub CI.
- Keep static validation of committed PNG dimensions, DPI, color profile,
  size limits, filenames, and README references in GitHub CI.
- Keep all documentation renderer source compiled by the test target.
- Preserve all unrelated application rendering tests, including surface and
  shadow behavior tests.

## Non-goals

- Do not remove the renderer, its local script, or its production-component
  seams from the repository.
- Do not weaken local two-pass determinism checks.
- Do not make generated images exempt from repository review.
- Do not change production application behavior or image content.
- Do not disable unrelated AppKit or SwiftUI tests.

## Design

### Explicit rendering-test boundary

Split the three tests that rasterize documentation views into a dedicated
`DocumentationPreviewRenderingTests` suite:

- `productionStatusViewKeepsQuotaTextAcrossRefreshHealth`
- `rendersApprovedAssets`
- `offscreenRenderingExercisesEveryInjectedSeam`

Keep the remaining fixture, state, settings-page, and isolated-store tests in
`DocumentationPreviewRendererTests`. Keep
`DocumentationPreviewSeamTests` unchanged.

Both rendering and non-rendering suites remain part of a normal local
`scripts/test.sh` run. The focused local generator filters
`DocumentationPreviewRenderingTests.rendersApprovedAssets`.

### GitHub CI contract

GitHub CI runs:

1. shell syntax checks
2. static committed-image validation
3. Swift tests with `DocumentationPreviewRenderingTests` skipped
4. source installation validation

GitHub CI no longer invokes `scripts/render-doc-previews.sh --check`.
Skipping the named suite is explicit in `.github/workflows/tests.yml`; no
environment-sensitive early return is hidden inside Swift tests.

The renderer source remains in the compiled test target. CI therefore still
catches Swift compilation failures in renderer code, but it does not create
documentation PNG data or open documentation-preview windows.

### Local contributor workflow

Contributors update README assets on the Mac that owns the approved visual
output:

```sh
scripts/render-doc-previews.sh
scripts/validate-doc-images.sh
```

The render script continues to generate into a temporary staging directory,
perform two renders inside `rendersApprovedAssets`, validate image metadata,
and install all four files with rollback protection. Contributors commit the
resulting PNG changes together with the corresponding README or source
changes.

## Failure Behavior

- A malformed or missing committed PNG fails GitHub CI.
- Incorrect dimensions, DPI, sRGB tagging, size limits, or README references
  fail GitHub CI.
- Ordinary Swift or installation regressions fail GitHub CI.
- Local nondeterministic documentation rendering fails the local generator.
- GitHub CI does not declare an approved local PNG stale based on a
  different macOS renderer.

## Tradeoff

GitHub CI cannot automatically detect that a production UI change requires a
fresh README render. That freshness decision becomes an explicit contributor
and review responsibility. This is preferable to treating platform-dependent
PNG bytes as a portable correctness signal.

## Verification

The change is complete when:

- the GitHub workflow contains no documentation render invocation
- the CI Swift command skips only
  `DocumentationPreviewRenderingTests`
- a local filtered CI-equivalent test run executes no documentation rendering
  tests
- a normal local `scripts/test.sh` run still executes all rendering tests
- `scripts/render-doc-previews.sh --check` passes locally
- `scripts/validate-doc-images.sh` passes
- `scripts/test-install.sh` passes
- the pushed GitHub Actions run passes through source installation
